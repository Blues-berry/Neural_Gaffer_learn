结合你当前代码和“已经接近满显存”这个现实，我会这样取舍你上面那三个方向：

1. 多尺度高光一致性
2. 局部相对亮度高光定义
3. latent-space highlight probe

我的结论先放前面：

- **最优先**：`局部相对亮度高光定义`
- **第二优先**：`latent-space highlight probe`
- **最后再做**：`多尺度高光一致性`

原因核心就一句话：你现在最缺的不是“更复杂的监督形态”，而是“在不继续加显存的前提下，把高光定义做得更准，或者把最贵的 image-space 分支搬走”。

**1. 局部相对亮度高光定义：最适合现在立刻做**
这是当前代码里最顺滑、最便宜、最不容易炸显存的方向。

你现在的高光流水线已经很完整了：
- 前景 mask
- quantile threshold
- blur 只用于 threshold 估计
- 最终 `normalized_excess / soft_score / mask_map` 仍基于原始 luminance

相关逻辑就在 [neural_gaffer_training.py](/4T/CXY/Neural_Gaffer/neural_gaffer_training.py#L788) 附近那套 `resolve_highlight_threshold_map` / `compute_highlight_weight_map` / `compute_highlight_mask`。

所以最自然的增强方式不是再加一条大支路，而是把“亮”从绝对亮度换成“相对局部背景更亮”：
- 例如 `relative_excess = luminance - local_mean`
- 或 `luminance / (local_mean + eps)`
- 再在前景内做 quantile threshold

这条路线的优点：
- 几乎不增加显存
- 改动集中在高光图生成函数里，不碰 UNet/VAE 主干
- 更容易避免把“大面积本来就亮的漫反射区域”误判为高光
- 很适合 area-light 的宽高光

如果你现在显存已经满，这条是**性价比最高**的创新点。

**2. Latent-space highlight probe：最值得做，但应放在第二步**
这是三者里最可能同时“省显存 + 提速 + 变成方法点”的方向，但它不是最便宜的第一步。

你当前最大的额外显存热点很明确，就是 image-space highlight loss：
在 [neural_gaffer_training.py](/4T/CXY/Neural_Gaffer/neural_gaffer_training.py#L1938) 开始，从 `pred_x0_latents` 走到 `decode_latents_to_image(...)`，再算 image loss。这个分支会让反传穿过 VAE decoder。

而且你当前配置确实开着这个分支，见 [neural_gaffer_training_gpu1_highlight.txt](/4T/CXY/Neural_Gaffer/configs/neural_gaffer_training_gpu1_highlight.txt)。

所以如果你的目标是“显存已经顶满了，还想往上走”，那 latent probe 的价值非常大，因为它是唯一一个有机会**把最贵的 decode 反传彻底拿掉**的方案。

但为什么我不把它排第一？
因为它虽然收益最大之一，但需要新训练流程：
- 先构造 latent-highlight 配对
- 训练一个 probe
- 再接回主训练
- 再验证它不会伤主指标

也就是说，它更像一个“二阶段升级”。  
如果你现在想快速继续推进实验，先做它会拉长迭代周期。

所以我的建议是：
- 如果你现在主要想**尽快出一版更强更稳的结果**，先别马上上 probe
- 如果你现在主要想**解决显存瓶颈并为论文加一个更硬的方法点**，它就是下一步最值得投入的

**3. 多尺度高光一致性：当前阶段最不划算**
这条从论文表述上很好听，但结合你现在的代码和显存状态，我会最后做。

原因有三个：

第一，它大概率会继续加显存。  
你现在如果在 `32x32 / 64x64 / 256x256` 多尺度都算高光监督，不管是多尺度 GT map、pred map，还是中间 feature 对齐，都会增加额外 tensor 和 loss 分支。你现在已经接近满显存，这条路和你的现实约束是冲突的。

第二，它对当前代码侵入更大。  
你现在的 highlight 监督主要围绕：
- latent diffusion per-pixel loss
- image-space decoded loss

如果上多尺度，你就得明确每个尺度到底监督谁：
- latent
- decoded image
- 或 UNet 中间 feature

这会从“改高光定义”变成“改训练结构”。

第三，它不直接解决当前最贵的点。  
你最贵的是 image-space decode 分支，不是“高光图太单尺度”。所以它可能增加复杂度，却不减核心负担。

因此，这条更适合：
- 显存不紧的时候
- 或你已经做完 latent probe、把 decode 分支拿掉以后
- 再作为论文增强版加入

**结合当前状态的最终取舍**
如果我是你，我会按这个顺序走：

1. **先做局部相对亮度高光定义**
   目标：几乎不增显存，看看主指标能不能继续涨，特别是高光区域 precision 能不能更稳。

2. **再做 latent-space highlight probe**
   目标：把 [neural_gaffer_training.py](/4T/CXY/Neural_Gaffer/neural_gaffer_training.py#L1938) 这条最贵分支替掉，真正释放显存预算。

3. **最后再考虑多尺度高光一致性**
   目标：在前两步稳定后，把方法包装得更完整，而不是现在就往显存上加负担。

**一句话总结**
在“已经满显存”的前提下：
- `局部相对亮度` 是最该先做的，因为便宜、稳、和现有代码最连续
- `latent probe` 是最值得投入的中期方案，因为它能真正换来显存和方法增量
- `多尺度一致性` 现在先别急，它更像显存宽裕后的增强项

如果你愿意，我下一步可以直接帮你把“局部相对亮度高光定义”落成一个最小改动方案，具体到应该改哪几个函数、加哪两个参数。

**2026-03-23 Update: Local Relative Highlight Definition**

这次在原有 `foreground mask + quantile threshold + blur only for threshold estimation + raw-luminance final score` 的高光流水线上，继续补上了“局部相对亮度”定义，目标是提升高光区域 precision，尤其是在大面积亮背景和宽高光波瓣下的稳定性。

当前实现已经接入训练、验证和可视化三条路径，核心逻辑在 [neural_gaffer_training.py](/4T/CXY/Neural_Gaffer/neural_gaffer_training.py#L791) 和 [neural_gaffer_training.py](/4T/CXY/Neural_Gaffer/neural_gaffer_training.py#L859)：
- `highlight_relative_mode = difference`
- `relative_excess = luminance - local_mean`
- `quantile threshold` 在 relative-measure 域里做
- `blur` 仍然只用于 threshold estimation
- 最终 `normalized_excess / soft_score / mask_map` 仍然基于原始 `luminance` 和最终 `threshold_map`

这次额外做了一个关键修正：`local_mean` 不再是简单全图 `avg_pool`，而是改成了前景内归一化局部均值。这样白背景和图像边界不会把局部参考亮度带偏，更符合“相对局部背景更亮”的高光定义，也不会明显增加显存开销。

当前实验配置见 [neural_gaffer_training_gpu1_highlight.txt](/4T/CXY/Neural_Gaffer/configs/neural_gaffer_training_gpu1_highlight.txt#L38)：
- `highlight_quantile = 0.88`
- `highlight_min_threshold = 0.02`
- `highlight_max_threshold = 0.2`
- `highlight_quantile_blur_sigma = 1.0`
- `highlight_relative_mode = difference`
- `highlight_local_kernel_size = 15`
- `highlight_relative_eps = 1e-4`
